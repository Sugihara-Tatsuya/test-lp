# クロスオーダー Brain モック プレイブック

## これは何か

「クロスオーダー Brain」は、クロスマート社の経営データ（MRR・受注・チャーン・プロダクトKPI・商談・経営会議）を集約して経営者に見せるAIエージェント製品のコンセプトで、2026-06-25の経営会議で事業化が正式決議されている（当初想定 月額50〜60万円、撤退基準 12ヶ月でMRR200万円未満）。

このリポジトリにあるのはその**単一HTMLモック（11画面）**で、Slack/Notion/Google Drive/GA4/ローカル議事録から集めた**実数値**で構成されている。つまりこれは「見た目のデモ」ではなく「数値が正しいことに価値がある資料」であり、経営会議にそのまま出せる精度を保つことが仕事の本質。見た目の作業より数値の裏取りに時間を使うのが正しい配分。

## 鉄則（重要な順）

1. **数値を創作しない。** モック上のすべての数値には出典（Slackチャネル・Notionページ・シート名＋日付）がある。新しい数値を置くときも同じ基準を守る。裏取りできない数値は「不明」と明記するか、置くなら「※推計」バッジ＋根拠の注記をつける。それらしい数字で埋めるくらいなら空欄のほうがよい — このモックの信用はそこで決まる。
   - **派生表示値（前年同月比など）にも同じ基準を適用する。** 例: 6月実績に更新したとき、前年6月の実データがないのに「前年5月基準から逆算した前年比」を再計算して置くのは創作。正しい基準期間のデータがなければ据え置くか注記・非表示にし、どう扱ったかを完成報告に必ず明記する。

2. **確定値と推計値を区別する。** 例: `BRAIN_DATA.actual.bookedMRR` には `bookedIsEstimate` フラグがあり、画面に※推計バッジが出る。推計を確定値で裏取りできたら値を差し替えてフラグを落とす。逆に確定値のフリをした推計を混ぜるのは最悪の失敗。

3. **単位と「どの月の数字か」に敏感になる。** このプロジェクトで実際に起きた事故がこの2種類:
   - **対象月ズレ**: Slack #all_board_management_meeting の月次予実報告は「報告月の前月」の実績（2026-06-09報告の10,683万円は5月実績）。`mrrSeries` は対象月ベース（`ym` は実績の月）で記録する。
   - **月次額と期間合計の混同**: 「チャーン▲384万円」は残り6ヶ月の合計で、月次は▲64万円。文章生成時に「/月」をつける前に必ずどちらか確認する。
   - 金額は万円/千円/円が混在するソースから来る。シートは千円単位のことが多い。転記時に必ず単位を書く。

4. **原本や既存構造を鵜呑みにしない。** 実例: 原本シートのQ目標「アップセル60/クロスセル70」は、調査の結果**社内に実在しない目標**だった（サクセスのKGIは合算「エクスパンションMRR 月130万円」）。数字が既存資料に書いてあることと、それが今も社内で使われている管理粒度であることは別。裏取りで食い違ったら、実際の管理粒度に合わせて構造ごと直し、暫定・仮置きのものは「暫定」「仮置き」バッジで正直に表示する。

5. **重要な前提は2系統で裏取りする。** 例: 課金開始ラグはNotion「FY2026達成に向けた論点整理」とSlackの岡林投稿の両方で確認してから確定扱いにした。1ソースしかない重要前提は「暫定」扱いで進め、モデルの結論に効く場合はその旨を明記する。

6. **機微情報を載せない。** 経営会議議事録には人事（退職者・採用候補・SO）、個社インシデント、資金繰りの生々しい情報が含まれる。モックに載せるのはビジネス数値中心、人名は姓のみ、個社の機微は一般化する。実装後、議事録由来の要注意ワード（人名・個社インシデント名など、扱った議事録に応じてリストアップ）で grep して残存0を確認する。

## 確定済みドメイン知識（再調査不要。ただし金額は変わるのでBRAIN_DATAが正）

- **会計年度は暦年**（12月期決算）。3Q=7〜9月、4Q=10〜12月。ただし社内には「6月始まりの営業評価サイクル」「7月〜翌6月の契約満了サイクル」の記述も併存する — 経営会議・皮算用の文脈では暦年を使う。社内隠語「肉」=4Q。
- **チャネル区分**: 新規獲得（FS）／バディ／アップセル／クロスセル／エンプラ。アップセル＋クロスセル＝「エクスパンション」（サクセス側の管理単位）。
- **「バディ」は二義語**: ①帳票変換プロダクト（新規チャネルの内数） ②社内のCS/カスタマーセールス2人一組体制の呼称。文脈で判別する。
- **課金開始ラグ**: 新規=受注の2ヶ月後、アップセル=当月（例外: 「無償→2ヶ月で自動アップセル」条項付きは2ヶ月）、クロスセル=2ヶ月後。したがって**10月以降の新規・クロスセル受注は年内MRRにほぼ効かない** — これが皮算用モデルの最重要構造で、「7〜9月受注の前倒しが決定的」という経営メッセージの根拠。旧モデル（単月×残月）がこれを無視して101.3%と楽観誤答した経緯がある。
- **チャーン**: 率ベース 月0.5%がデフォルト（2026年6月実績56.1万円≒0.53%で校正。FY2025実績0.4%）。率/固定額のトグルあり。
- **FY2026計画**: 年末MRR 13,073万円。
- **報告サイクル**: 月次予実（課金MRR実績）は翌月第2週頃（例: 6月実績は7/8頃）に #all_board_management_meeting へ。チャネル別受注の月締めは #revenue-team。

## BRAIN_DATA 設計原則

- 全数値はファイル冒頭の **`BRAIN_DATA` 単一オブジェクト**にのみ定義。全11画面がここから描画される。数値変更は原則ここだけを触る。
- **値を変えたら対応する `asOf` / `*AsOf` も必ず更新**する。as-ofバッジは自動描画され、90日超で黄警告になる。asOfの更新漏れは「古い数字を新しいと誤認させる」ので値の間違いと同罪。
- 皮算用モデルは **`simulate()` 純関数**（`window.__simulate` にexpose）。シナリオ×チャネル×受注月の受注グリッド＋課金ラグ＋月次シミュレーション（チャーン率連動・料金改定は10月一括ステップ）。
- **セルフチェック機構**: `runSelfCheck()` が `console.assert` で期待着地・達成率を検証し、皮算用画面の `#checkBox` に「全項目OK」を表示する。モデルの前提値を恒久変更したときは、Consoleから `window.__simulate` を直接叩いて期待値を再計算し、`expectedLanding` / `expectedRate` を更新する（期待値の更新を忘れるとassertが恒久的に落ちる）。
- 画面上の編集セルは一時的なシミュレーション用途（リロードで消える）。恒久変更は必ず BRAIN_DATA 側を書き換える。

## 作業の型

### 1. 調査（数値集め）

どの数値がどこにあるかは **`references/data-sources.md` を必ず読む**（Slackチャネル・Notionページ・Drive fileId・GA4プロパティの対応表）。

データ収集をサブエージェントに切り出す場合の指示のコツ（実績のある型）:
- ソース別（Slack/Notion/Drive/ローカル/GA4）に1体ずつ分け、並列で走らせる
- 検索・閲覧の**打ち切り上限を明示**する（「検索20回まで」「ページ10件まで」）— 上限がないと深追いして帰ってこない
- **出典＋単位を必須**にし、見つからない項目は「不明」と明記させる（推測で埋めることを明示的に禁止する）
- 最終メッセージは「前置き・挨拶不要、構造化Markdownのみ」と指定（機械的に使うため）

### 2. 設計（実装前に数値を確定する）

- 画面仕様と**数値仕様（全数値＋出典＋計算式＋期待値）を別ファイルで先に書き**、それをビルドの「正」とする。数値と実装を分離すると、ビルダーの創作を防げて監査もしやすい。
- モデルや前提を変えるときは、**実装前に期待値（着地・達成率）を自分で手計算**しておく。実装後の検証はこの手計算と `__simulate` の突き合わせで行う。

### 3. 実装

- 技術制約: Tailwind CDN + Google Fonts（Noto Sans JP / Outfit）+ インラインSVGのみ。他の外部ライブラリ禁止。vanilla JS。数値は `.num` クラス（tabular-nums）＋3桁カンマ、マイナスは赤字▲。
- **v78デザイン参照の読み方（重要）**: 1.1MBあるので全読み禁止。`grep -n "探したいUI文言" ファイル` で該当行を見つけ、周辺±60〜80行だけReadして className をそのまま流用する。v78はReactだがモックはvanilla JSでよく、**見た目のクラス構成の一致が最優先**。
- 既存の皮算用モデル（simulate / console.assert / 受注グリッド / トルネード）は、その変更が目的の作業以外では**一切触らない**。

## 経営分析・サマリを求められたら

数値はモック（BRAIN_DATA）と整合させ、出典を添える。効果的だった構成:

1. **結論を1段落**（計画比どうか＋挽回レバーは何か）
2. 好調な点 / 懸念点（それぞれ数値つき箇条書き）
3. シナリオ表（成行/改善/さらに改善 × 前提・着地・達成率）
4. 計画到達へのレバー（金額インパクト順。感度分析トルネードの値を使う）

分析の視点として持っておくこと: MRR残高が計画通りでも受注（獲得）が未達なら、それは「過去の貯金」であり数ヶ月後に残高に響く。残高と獲得を分けて評価する。
